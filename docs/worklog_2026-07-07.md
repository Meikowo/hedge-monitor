## 二、本轮体检发现的问题（按严重程度）

### P0 —— 不修会直接卡住 M1/M2

1. **Kimi 版 `companies_data_all.sql` 不建议原样执行。** 语句结构本身没错（5527 行、无重复主键、无引号转义问题、12 条 INSERT），但有三处实质缺陷：
   - **没有 `ON CONFLICT`**：`sec_code` 是主键，第一次能跑通，但**任何重跑/断点续跑都会整条语句报错回滚**，不幂等；
   - **沪市 1698 家 + 科创板约 600 家的行业是空字符串 `''`**（不是 NULL）——41% 的公司没有行业，且 `''` 在筛选/分组里会变成一个假类别；
   - **行业口径不一致**：沪深是证监会门类（`C 制造业`），北交所却是细分行业（`医药制造业`），同一列两套口径，「分行业」维度直接失真。
   → 修复版 `supabase/10_companies_data_fixed.sql`：`''`→NULL、北交所归一到门类、每条语句带「补空不覆盖」的 ON CONFLICT，可无限次安全重跑。
2. **`extract_pipeline.py` 的取数逻辑撑不住回填规模。** 原 `fetch_pending` 用 `.limit(limit*3)` 一次取数，受 Supabase PostgREST **单次 1000 行上限**约束：2025 年回填入库后，未抽取积压排在几千行深处，`--limit` 给再大也取不满、极端情况取不到。→ 已改为按发布时间倒序深分页遍历，`--limit 1000` 也能正确凑满。**这个不修，「清零 2025 积压」这一步会莫名其妙抽不动。**

### P1 —— 不修会持续埋雷

3. **`build_companies.py`（Kimi 版）三个问题：** ① 行业只来自深交所/北交所接口，沪市天然拿不到（这就是 SQL 里 41% 空行业的根因）；② `on conflict do nothing`——表里已有行永远不更新，先跑 SQL 后这脚本形同虚设；③ `list_date` 传的是 `date` 对象，supabase-py 序列化会抛 `not JSON serializable`。→ v2 重写：用**东方财富行业板块（约 86 个一级行业）给全市场统一打标**（粒度远优于「C 制造业占全市场四成」的门类），证监会行业兜底；upsert 改为「带行业的行覆盖更新、无行业的行不携带行业列」，物理上不可能把已有值洗掉；日期转 ISO 字符串；新增 `--emit-sql`。**注意：`ent_type`（企业性质）仍无免费可靠数据源**，脚本刻意不碰这一列——真实路径见第五节 M1 遗留项。
4. **`need_review` 判定会淹没复核队列。** 制度/可行性/进展类公告（`is_hedging_announcement=false`）本来就没有额度，原逻辑却因 `limit is None` 全部打回复核——回填几千条后队列全是非事件。→ `extract_core.py` 已修：仅当实质公告缺额度才复核。
5. **backfill workflow 没有并发锁。** 手滑同时触发两个年份 = 双倍请求轰巨潮，极易触发风控（worklog 已知坑第 1 条）。→ 加 `concurrency: cninfo-backfill`（排队不取消）。
6. **没有 `.gitignore`。** `.env` 里是 service role key（数据库全权限），`data/` 是 PDF 堆。→ 已补。

### P2 —— 顺手清掉

7. `docs/` 两份相同 PRD → 删中文名那份，只留 `docs/PRD.md`（和当初两个 index.html 同一类病）。
8. `.env.example` 缺 LLM 变量 → 已补 `LLM_API_KEY / LLM_BASE_URL / LLM_MODEL`。
9. 新增 `.github/workflows/build-companies.yml`：Actions 里一键刷新公司维表，不需要本地 Python；akshare 只在该 workflow 安装，不拖慢日常任务。

**没发现问题、无需动的：** 爬虫（分页/限速/排除规则健全）、回填脚本主体（季度切窗+幂等正确）、入库脚本、抽取提示词与 JSON 容错、schema.sql、02/03 迁移、日更与抽取 cron。

**第 3 步 · 验证 2025 回填结果（Supabase，1 分钟）**
```sql
select count(*), min(publish_time), max(publish_time)
from announcements
where publish_time >= '2025-01-01' and publish_time < '2026-01-01';
```
把这个数发我。经验量级应在**大几百到两三千**；若明显偏低（比如只有一两百），说明有季度窗被巨潮风控吞了——直接在 Actions 重跑一次 2025（幂等，不会重复入库）。

**第 4 步 · 刷新全市场行业（Actions，挂机约 5 分钟）**
GitHub → Actions → **Build Companies Dimension** → Run workflow（dry_run 留 false）。
验收：`select count(*) from companies where sw_industry is not null;` → 应升到 **5300+**，且行业变成「有色金属 / 化学制品」这类细分口径。
（若 Actions 海外 IP 拉东财失败，本地跑 `python scripts/build_companies.py` 等效；再不行加 `--skip-em` 至少保住证监会口径。）

**第 5 步 · 清 2025 抽取积压（Actions，挂机数小时，花 LLM 钱）**
Actions → **Extract Hedging Fields (LLM)** → Run workflow，`limit` 填 **500**，since 留空。跑完看日志末尾「成功/失败」，再点下一轮，直到日志显示「待抽取公告 0 条」。
监控积压：
```sql
select count(*) from announcements a
left join hedge_events e on e.announcement_id = a.announcement_id
where a.is_candidate and a.pdf_url <> '' and (e.extracted_at is null);
```
每轮 500 条约 1–2 小时、MiniMax 费用很低；别一次填几千（单次 6 小时 timeout 可能不够）。

**第 6 步 · 逐年回填 2024→2021（Actions，每年挂机一次）**
Actions → **Backfill Announcements (Historical)** → year 填 2024 → 跑完→抽取清零（重复第 5 步）→ 再下一年。并发锁已加，就算手滑连点也只会排队。
