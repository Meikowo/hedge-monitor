# Worklog — 2026-07-21（前端加载修复）

## 问题

Pages 更新后，页面长期停留在“正在读取结构化数据”，15 分钟仍不结束，也没有进入错误状态。

## 定位

- Supabase 数据、匿名读取权限、线上公开配置和 CORS 均正常。
- 原启动流程为了顶部一个“结构化公告”数字，完整分页下载全部公告状态：当前 8,000 多行、9 次请求。
- 加上事件层 4 次分页，首次打开约需 13 次请求；原生 `fetch` 没有超时和重试，任意请求长期 pending 都会让静态转圈永久保留。
- 新增省份本身只增加很少传输量，不是 15 分钟卡住的原因。

## 修复

1. 顶部公告数字改用 `HEAD + Prefer: count=exact`，只读取响应头中的准确总数。
2. 完整公告工作区保持原逻辑：进入“公告原流”后仍分页读取全部公告、证据、详情和导出字段。
3. 所有 Supabase 请求增加 20 秒超时；网络错误、429 和临时 5xx 最多尝试 3 次。
4. 转圈下方显示事件/公告已读取条数和重试状态。
5. 增加全局脚本与 Promise 异常处理，初始化失败时显示可读错误，不再无限转圈。
6. 删除事件/公告请求中从未展示的字段，事件完整载荷由约 3.35 MB 降至约 2.91 MB。
7. HTML 为 CSS、公开配置和 JavaScript 增加 `v=20260721-2`，避免 Pages/CDN 缓存新旧文件错配。

## 验证

- `node --check web/app.js`：通过。
- `tests/test_web_structure.mjs`：通过，46 个前端引用节点完整。
- `tests/test_web_bootstrap.mjs`：使用公开只读配置真实启动通过，约 5.4 秒完成；读取 3,281 个事件和 6,763 条已结构化公告。
- 轻量计数响应为 HTTP 206，`Content-Range` 正确且已通过 CORS 暴露。

## 影响边界

- 不改数据库、不改 RLS、不改抽取数据。
- 不减少事件表或公告表中的最终展示字段。
- 不增加 LLM 调用和费用。
