## 1. 北极星（项目最终要回答的问题）

一句话：**既能从全局看清「A 股上市公司整体如何参与套期保值与期货市场」，又能下钻到单家公司，详细地回顾、了解、分析、跟踪它套保的额度、品种、场所、审批，以及计划与实际效果的差距。**

拆成七件事（原始需求，第 7 条为 v1.2 新增）：

1. 每天监控套保**公告**的发布
2. 解析**年报**里披露的套保业务情况
3. 既看**概览**（参与全貌、趋势、分布），又看**细节**（额度、品种、场所、审批层级）
4. 能**分行业**查
5. 能**分公司类型**（央企 / 地方国企 / 民企 / 外资）查
6. **日更** + **可查 5 年历史**
7. **（v1.2）** 可查每家公司**近 5 年半年报与年报**中披露的套保**业务规模与损益**；每一期都如实标注「有数值 / 提及但无数值 / 未提及」，**绝不虚构或推算数值**

**核心价值：** 把公告里的「计划」（额度、品种、工具、场所）与定期报告里的「实际」（业务规模、品种、**报告期实际损益**）对起来，做**品种核对、模式核对、数值核对**三维的「计划 vs 实际」套保效果分析。这是明确要做、且市面极少有人做好的部分，也是本系统的最终目的地。

## 2. 用户与场景

| 项 | 结论 |
|---|---|
| 用户 | 期货研究所研究员（你本人） |
| 用途 | 服务**风险管理业务**：详细回顾、了解、分析、**跟踪**上市公司套保的具体情况（公告侧日更 + 定期报告侧半年度/年度双频观测） |
| 双重定位 | ① **研究**：分析上市公司套保行为与效果；② **业务线索**：识别谁在套保、规模多大、做什么品种、用境内还是境外——直接关系风险管理子公司的展业方向 |
| 是否公开 | 否（很长一段时间内自用） |
| 是否登录 | **否** |

**由此确定的约束与要求：**

- 不做用户系统、不做鉴权；Supabase 保持匿名只读（RLS）即可；前端可保持**纯静态页面**直接读 Supabase，无需后端服务器、无需 ICP 备案与 SEO。
- 因为产出会用于**专业研究判断与对客展业**，**数据准确性是硬要求**：抽取结果必须可核验（保留置信度与证据链），关键字段（额度口径、境内外、品种、损益）需定期人工抽检。
- **（v1.2 强化）找不到就是找不到**：任何数值抽不出来时，系统各层（抽取、入库、展示）必须如实呈现「未披露 / 未找到」，严禁编造、推断、换算补全，也严禁把空值显示为 0。具体机制见 5.7。
- **（v1.2 更新）**界面要求「**克制的设计感**」：优雅、不堆砌装饰，但**信息密度与可追溯性**优先——研究员要能快速从一条线索钻到原文。具体设计语言见 7.6，作为 M3 前端重做的设计任务书。

## 3. 范围边界（Scope）

### 3.1 要做（In Scope）

- 套保**公告**采集：商品 / 外汇 / 利率三类全收
- 公告**结构化抽取**：额度、品种、场所、工具、审批、额度口径、循环额度、自有资金等
- **公司维表**：行业、公司类型、板块（支撑分类查询）
- **定期报告（年报 + 半年报）**套保业务采集与解析：业务规模、品种、**报告期实际损益**，回溯近 5 个完整会计年度（纳入规则见 4.4）
- **计划 vs 实际三维匹配核对**：品种核对、模式核对、数值核对（核心价值，规范见 5.6）
- **前端**：概览 Dashboard + 公告流 + 公司详情（含规模与损益时间线、核对卡）+ 关注列表
- **5 年**历史、**日更**、**CSV 导出**

### 3.2 暂不做（Out of Scope）

- 登录 / 多用户 / 权限
- 公开运营 / SEO / 备案
- 秒级实时（公告是离散事件，无意义）
- 推送通知（「谁发新套保就通知我」）——列为后期可选
- 商用 / 数据转售（若将来要做，需另走数据授权合规）
- 全市场定期报告扫描（只做公司池，见 4.4；季报也暂不做——衍生品披露主要在半年报与年报）

## 4. 数据覆盖与纳入规则

### 4.1 套保类别（全收，且可分类看）

- **商品**（铜铝锌镍、原油、农产品、生猪等）
- **外汇**（远期结售汇、外汇掉期、外汇期权等）
- **利率**（利率互换等）

### 4.2 公告纳入规则

- 以 **《开展 XX 年度套期保值业务》** 类实质公告为主体（由 LLM 判 `is_hedging_announcement=true`）。
- **进展、平仓**类公告：收录，挂到同公司的时间线上，不当作新计划。
- **制度、可行性分析报告**类：标注但**不混入**实质业务数据（`is_hedging_announcement=false`）。
- **董事会决议**类（标题不含套保关键词、正文藏有套保议案）：走正文二次判别后收录。
- **纯投机性衍生品**（非套保目的）：标注区分，不混入套保统计。

### 4.3 排除规则

- 上市**期货公司自身**的公告（瑞达、南华、永安、弘业等，按代码白名单排除）。
- 股权激励噪音（标题含「期权激励 / 限制性股票 / 股票期权」）。

### 4.4 定期报告纳入规则（v1.2 新增）

- **报告类型：** 年度报告 + 半年度报告。排除摘要版、英文版；同一期存在「更正后」版本时以最新版为准（沿用 `is_revised` 逻辑）。季报暂不纳入。
- **公司池（控制体量与成本）：** 只采集「公告库 `hedge_events` 中出现过套保事件的公司 ∪ 关注列表公司」的定期报告，**不做全市场**。理由：本系统的定期报告解析服务于「计划 vs 实际」与个股跟踪，主体天然是已知套保公司。已知局限：极少数从不发套保公告、只在定期报告中披露衍生品的公司会漏，接受此损失；后续可按行业白名单扩池。
- **时间窗：** 近 5 个完整会计年度（当前即 2021–2025 年度）+ 当年已披露期 → 每公司约 10–11 期。
- **解析落点（按优先级）：**
  1. 正文「**衍生品投资情况**」表：初始投资金额、报告期内购入/售出金额、计提减值（如有）、期末金额、期末占公司净资产比例、**报告期实际损益**——业务规模与损益的主来源；
  2. 财务报表附注：衍生金融资产/负债、套期会计（公允价值套期 / 现金流量套期）、其他综合收益中的套期储备；
  3. MD&A / 重要事项中的定性描述（品种、目的、境内外）。
- **三态披露状态（一等公民字段）：** 每公司每期必须落一条记录，`disclosure_status` 取值：
  - `disclosed_with_numbers`（有表有数）
  - `mentioned_no_numbers`（提及套保/衍生品但抽不出可靠数值）
  - `not_mentioned`（全文未提及）
  三态都**如实入库、如实展示**。「查了但没有」和「还没查」是两种不同状态，不得混淆。

## 5. 数据模型（核心表）

> 现状说明：仓库 `schema.sql` 已建 `announcements`、`annual_reports`（仅元数据）、`tips`、`hedge_events`、`extraction_evidence` 及视图 `v_announcements_with_events`。**缺 `companies` 表；且抽取扩展列存在于线上库但未提交进仓库 schema（见第 10 节技术债）。**

### 5.1 公司维表 `companies`（待建）

代码、简称、orgId、申万三级行业、板块（主板 / 创业板 / 科创板 / 北交所）、企业性质（央企 / 地方国企 / 民企 / 外资 / 其他）、市值、上市日期。
> 「分行业、分公司类型」两个查询的**唯一数据载体**，也是展业筛选（按行业、按性质找客户）的基础，同时是 4.4 公司池的判定依据之一。

### 5.2 公告事实表 `announcements`

announcementId（主键、去重键）、证券代码、简称、标题、披露时间（北京时区）、公告分类、PDF 直链。

### 5.3 套保业务明细 `hedge_events`（从公告抽取）

- 套保类别 `hedge_type`（商品 / 外汇 / 利率 / 其他衍生品）
- 风险类型 `risk_type[]`、交易工具 `instrument_type[]`（期货 / 期权 / 远期 / 掉期 / 互换）
- 具体品种 / 标的 / 币种 `underlying_asset[]`
- **额度 `contract_value_limit` + 额度口径 `contract_value_basis`**（保证金最高占用额 / 业务总额 / 合约价值）+ 币种 `contract_value_currency` + 原文 `contract_value_raw_text`
- **交易场所 `trade_venue`**（境内 / 境外 / 境内外；LME、CME、芝商所属境外）
- 审批层级 `approval_level`（董事会 / 股东大会 / 两者）、业务期限 `authorization_period`
- 是否循环额度 `is_revolving`、是否自有资金 `use_own_funds`、是否实质公告 `is_hedging_announcement`
- 置信度 `confidence`、复核标记 `need_review`、抽取时间 `extracted_at`、PDF 留档路径 `pdf_storage_path`

> ⚠️ **额度口径陷阱（贯穿全系统的设计纪律）：** 商品套保金额通常是「保证金最高占用额」（千万级），外汇是「业务总额」（千万美元级），含义与量级完全不同，**严禁跨类别直接求和或同榜排名**。所有汇总、排行必须**分口径**处理。该纪律同样约束 5.6 的数值核对。

### 5.4 证据表 `extraction_evidence`

字段名、页码、原文引用句、置信度——支撑「从一条结论一键钻到原文」的可追溯性。公告与定期报告两条抽取管线共用。

### 5.5 定期报告衍生品披露表 `periodic_derivatives`（待建 — 最硬，最后做；v1.2 由 annual_derivatives 扩展而来）

- 关联与期别：`report_id`（关联定期报告元数据）、证券代码、报告类型 `report_type`（年报 / 半年报）、报告期 `report_period`（如 `2025FY` / `2025H1`）
- **披露状态 `disclosure_status`**（三态，见 4.4——本表最重要的字段之一）
- 品种/标的 `underlying_asset[]`、工具、目的 `purpose`（套期保值 / 投机 / 未说明）
- 数值组（**币种与单位必须随原文记录**）：初始投资金额 `initial_investment`、报告期购入 / 售出金额、**报告期实际损益 `period_pnl`**、期末金额 `ending_balance`、期末占净资产比例 `pct_net_assets`、套保比例（若披露）
- 会计处理 `hedge_accounting`（是否适用套期会计及类型，若披露）
- 质量组：**`amount_verified`（数值程序回验是否通过，见 5.7）**、`confidence`、`need_review`、证据（页码 + 原文句，入 5.4）、`source_section`（数值来自「衍生品投资情况」表 / 附注 / MD&A）

> 「报告期实际损益」与「期末金额（业务规模）」就是「计划 vs 实际」里**实际**的那一侧；半年报提供年中观测点，把跟踪粒度从年度提升到半年度。
> 元数据侧：现有 `annual_reports` 表增加 `report_type` 列扩展为定期报告元数据表（迁移随 M4a 提交进仓库）。
> 实现纪律：整本报告**严禁整篇直送 LLM**。先用目录/关键词定位「衍生品」「套期保值」「远期结售汇」等相关页，只把相关页送抽取——同时控制成本与幻觉面。

### 5.6 计划 vs 实际匹配核对（v1.2 新增 — 核心价值的落地规范）

- **匹配键：** 公司 + 会计年度 + 套保类别 `hedge_type`。公告授权期跨年（如「自股东大会通过之日起 12 个月」）时，按覆盖期间归入相关年度并标记 `cross_year=true`。
- **三维核对：**
  1. **品种核对：** 计划 `underlying_asset` 集合 vs 实际披露品种集合 → 输出交集、「计划有实际无」、「实际有计划无」；
  2. **模式核对：** 工具（期货 / 期权 / 远期 / 掉期）与场所（境内 / 境外）的计划 vs 实际；
  3. **数值核对：** **同口径才比**。实际「期末金额 / 累计投入」对计划「额度上限」仅在口径可比时计算占用率；商品「保证金最高占用额」口径与年报「合约名义金额 / 投资金额」不可直接比时，如实输出「口径不可比」而非硬算；**报告期实际损益单独展示，不与额度做大小比较**（一个是敞口上限，一个是盈亏结果，量纲不同）。
- **输出 `match_status`：** 匹配 / 部分匹配 / **仅有计划**（发了公告但报告未披露——可能未开展也可能未披露，如实标注，不做臆断）/ **仅有实际**（报告披露但无对应公告）/ 口径不可比。
- 「仅有计划」「仅有实际」本身就是有研究与展业价值的信号，是产出而不是脏数据。

### 5.7 抽取真实性纪律（v1.2 新增 — 防幻觉硬约束）

适用于公告与定期报告两条抽取管线；定期报告尤甚（长文档、表格复杂，是幻觉高发区）。四层防线：

1. **提示词层：** 明确要求「数值必须是原文中出现的数字；找不到就输出 null 并给出 `disclosure_status`；禁止推断、换算、估算、补全」；每个数值字段必须附证据（页码 + ≤30 字原文摘录）。
2. **程序回验层（关键兜底）：** 入库前将每个抽取数值归一化（去千分位、处理万/亿单位）后，与 PDF 原文文本做**字符串回验**；回验失败 → `amount_verified=false` + `need_review=true`，**不进入任何汇总、图表与匹配核对**，只进复核队列。防幻觉不能只靠嘱咐模型，要靠这层机制兜底。
3. **展示层：** `null ≠ 0`。未披露如实显示「未披露」；未披露的期**不参与**求和、均值、排行；每个数字可一键展开证据句与页码。
4. **抽检层：** 每批人工抽检 ≥20 条或 5%（取大者），记录字段级准确率；关键字段（损益、期末金额、额度口径）错误率超阈值即停线修提示词，**不带病扩量**。

## 7. 前端信息架构（设计提案，可调）

> 整体三层：**全局概览 → 公告明细 → 个股下钻**，外加对比页与关注列表。设计以「研究员能快速从线索钻到原文」为导向。

### 7.1 首页：概览 Dashboard

### 7.6 设计语言（v1.2 新增 — M3 的设计任务书）

**主题定位：「研报纸感的数据终端」。** 本站的气质应像一份排版考究的期货研究所内部研报，而不是通用后台管理模板（Bootstrap/AntD 默认感）也不是炫技大屏。所有视觉决策围绕本系统的两个本质：**信息密度**与**可追溯性**。

- **基调（用户点名可用 Claude 经典风格）：** 暖纸白底（近 `#F5F3EE`）、墨色正文（近 `#1F1E1C`）、深赭陶土色（近 `#C15F3C`）作为唯一的品牌/交互强调色，克制使用。卡片留白慷慨、分隔用细发丝线、圆角小而一致。
- **数据语义色与装饰色严格分离：** 损益遵循**中国市场惯例——红涨（盈）绿跌（亏）**；商品 / 外汇 / 利率三个类别各绑定一个固定色相，在全站所有图表与标签中保持一致编码，不许一图一套色。ECharts 必须套用与上述 token 对齐的自定义主题，禁用默认彩虹色板。
- **字体三角色：** 显示层用中文衬线（思源宋体/Noto Serif SC，标题与 KPI 大数）承担气质；正文用系统黑体保证密度下的可读性；**所有金额、代码、日期用等宽/表格数字（tabular-nums）**，右对齐，小数位对齐——这是数据产品的基本尊严。
- **签名元素：「证据引文卡」。** 全站任何一个抽取出来的数字或结论，点击即展开一张引文样式的小卡：原文句 + 页码 + 置信度 + 跳转 PDF。把「可追溯」这个系统灵魂做成最有记忆点的交互。「未披露」空档同样有专门的视觉记号（虚线占位 + 三态标签），与 0 在视觉上不可混淆。
- **质量底线：** 响应式到移动端、键盘焦点可见、动效克制（数据加载与证据卡展开可有轻微过渡，拒绝满屏动画）。
- **执行方式：** M3 在 Claude Code 中实施时，先按 frontend-design 技能流程产出 token 方案（色板 / 字级 / 布局线框 / 签名元素）自评一轮，再落码；本节为设计边界，细节允许在此边界内发挥。

**设计纪律：** 所有额度类视图分口径；未披露不显示为 0；概览以「回答问题/给线索」为导向，不堆砌数字。

## 12. 开工前盘点（数据量，需你在 Supabase SQL Editor 跑）

select min(publish_time), max(publish_time) from announcements;   -- 时间覆盖
